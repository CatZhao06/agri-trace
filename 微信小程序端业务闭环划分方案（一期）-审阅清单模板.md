# 微信小程序端业务闭环划分方案（一期）- 审阅清单模板

> 用途：用于快速检查“一期方案”是否具备可落地、可验收、可复盘的闭环设计。  
> 使用方式：每一项补齐后将状态改为 `[x]`，并填写负责人和截止时间。

---

## 1. 一期范围定义（Scope）

### 1.1 业务目标（必须量化）
- [ ] 明确一期主目标（示例：提升消费者扫码溯源成功率、建立防伪验证闭环）
- [ ] 目标有量化指标（示例：首扫成功率 >= 98%，查询 P95 < 1.5s）
- [ ] 目标有时间窗口（示例：上线后 30 天内达成）

### 1.2 范围内（In Scope）
- [ ] 消费者扫码进入小程序并完成溯源查询
- [ ] 防伪码验证与结果反馈
- [ ] 扫码日志采集、统计看板最小可用
- [ ] 异常场景基础兜底（弱网、码不存在、链路超时）

### 1.3 范围外（Out of Scope）
- [ ] 明确一期不做项（示例：售后工单、客服系统联动、深度风控）
- [ ] 明确与二期关系（是否保留接口、数据结构兼容策略）
- [ ] 明确临时人工处理方案（一期未覆盖但必须可运营）

---

## 2. 端到端业务闭环（泳道）

## 2.1 角色泳道
- [ ] 消费者（扫码、查看、验证）
- [ ] 小程序前端（交互、校验、重试、埋点）
- [ ] 后端服务（鉴权、查询、业务编排）
- [ ] 数据层/区块链（主数据、链上查询）
- [ ] 运营/审核（异常处理、数据复核）

## 2.2 闭环定义（强制填写）
- [ ] 起点事件：`__________`
- [ ] 终点事件：`__________`
- [ ] 成功闭环条件：`__________`
- [ ] 失败闭环条件：`__________`
- [ ] 闭环时限（SLA）：`__________`

## 2.3 关键路径（主流程）
- [ ] 扫码进入 -> code 解析 -> 查询码信息 -> 查询批次/产品/环节 -> 返回展示
- [ ] 防伪验证 -> antiCode 比对 -> 结果展示 -> 风险标记（如失败次数）
- [ ] 统计更新 -> scan_num/first_scan_time -> 扫码日志落库 -> 指标汇总

---

## 3. 状态机与流转表（必须有）

## 3.1 对象一：码申请状态（示例）

| 当前状态 | 触发动作 | 下一个状态 | 触发角色 | 可逆性 | 幂等要求 | 备注 |
|---|---|---|---|---|---|---|
| INIT(待审) | 审核通过 | APPROVED(通过) | 审核员 | 否 | 是 | 触发生码 |
| INIT(待审) | 审核拒绝 | REJECTED(拒绝) | 审核员 | 否 | 是 | 记录原因 |
| APPROVED | 生成码完成 | GENERATED(已生成) | 系统 | 否 | 是 | 失败可重试 |

检查项：
- [ ] 所有状态转移均有触发条件
- [ ] 非法状态跳转有拦截规则
- [ ] 并发重复提交有幂等键设计
- [ ] 超时/中断有补偿策略

## 3.2 对象二：防伪验证结果状态（示例）

| 场景 | 返回码 | 前端文案 | 是否记风险日志 | 是否触发告警 |
|---|---|---|---|---|
| 验证成功 | 0 | 正品确认 | 否 | 否 |
| 防伪码错误 | 1001 | 防伪码错误 | 是 | 视阈值 |
| 溯源码不存在 | 1002 | 溯源码不存在 | 是 | 是 |
| 系统异常 | 5000 | 系统繁忙，请稍后重试 | 是 | 是 |

## 4. 异常处理矩阵（一期最关键）

| 异常类型 | 发现方式 | 用户提示 | 系统动作 | 人工介入 | 恢复策略 | 负责人 |
|---|---|---|---|---|---|---|
| 码不存在 | 查询返回空 | 提示无效码 | 记录日志+风控计数 | 否 | 无 | 后端 |
| 防伪码不匹配 | antiCode 比对失败 | 提示疑似假冒 | 标记高风险 | 可选 | 复核批次 | 运营 |
| 弱网超时 | 前端请求超时 | 提示重试 | 指数退避重试 | 否 | 最多 N 次 | 前端 |
| 区块链查询失败 | SDK 异常 | 降级为基础信息 | 错误上报 | 是 | 异步补偿 | 后端 |
| 重复扫码高频 | 频控规则命中 | 提示频繁操作 | 限流/验证码 | 否 | 冷却时间 | 风控 |

检查项：
- [ ] 每类异常都有“用户提示 + 系统动作 + 责任人”
- [ ] 避免仅返回技术错误，统一业务错误码
- [ ] 支持问题追踪（traceId / requestId）

---

## 5. 数据与埋点口径（可运营）

## 5.1 必备埋点事件
- [ ] `scan_enter`（扫码进入）
- [ ] `trace_query_success`（溯源成功）
- [ ] `trace_query_fail`（溯源失败）
- [ ] `anti_check_success`（防伪成功）
- [ ] `anti_check_fail`（防伪失败）

## 5.2 埋点字段最小集
- [ ] code（脱敏或哈希）
- [ ] batchId
- [ ] traceType
- [ ] userId/openId（合规处理）
- [ ] networkType / deviceType / appVersion
- [ ] requestId / traceId

## 5.3 数据质量
- [ ] 去重策略（同一用户短时间重复上报）
- [ ] 丢数监控（埋点到达率）
- [ ] 字段字典和口径文档同步

---

## 6. 安全与合规（小程序场景）

- [ ] 敏感字段最小化采集（经纬度、用户标识）
- [ ] 位置授权被拒绝时可降级
- [ ] 接口频控与防刷（IP、设备、账号维度）
- [ ] 防重放（签名、时间戳、nonce）
- [ ] 日志脱敏与数据保留周期定义

---

## 7. KPI 与一期验收标准（可上线评审）

## 7.1 核心 KPI（示例）
- [ ] 首次扫码成功率 >= 98%
- [ ] 溯源查询接口 P95 < 1.5s
- [ ] 防伪验证正确识别率 >= 99.5%
- [ ] 异常闭环时长（含人工）< 24h
- [ ] 线上 Sev-1 事故 = 0

## 7.2 验收清单
- [ ] 主流程联调通过（全链路）
- [ ] 异常路径回归通过（至少 10 类）
- [ ] 监控告警可用（错误率、超时率、QPS）
- [ ] 运维预案和回滚方案可执行
- [ ] 业务方签字确认（产品/运营/技术）

## 8. RACI 责任矩阵（建议必填）

| 任务 | R(负责执行) | A(最终负责) | C(协作) | I(知会) |
|---|---|---|---|---|
| 需求口径确认 | 产品 | 业务负责人 | 运营/技术 | 全体 |
| 接口与状态机设计 | 后端 | 技术负责人 | 前端/测试 | 产品 |
| 小程序交互与异常提示 | 前端 | 前端负责人 | 产品/测试 | 运营 |
| 埋点与看板 | 数据/前端 | 产品 | 后端/运营 | 管理层 |
| 上线与回滚 | 运维/后端 | 技术负责人 | 前端/测试 | 业务 |
